Ticket Goal:
Replace startRandomInterval with a deterministic, auditable cron-driven scheduler

Problem

The current scheduler in server/src/jobs/scheduler.ts relies on startRandomInterval — a setTimeout-based utility that runs jobs at random intervals. This causes several issues:

  • Non-deterministic timing — You cannot predict when articles, recipes, or news-fetching will run
  • No visibility — There is no central registry of scheduled jobs; each job is scattered across files
  • No audit trail — If a job fails or is skipped, there is no log of what was supposed to run and when
  • Hard to debug — Reproducing timing issues requires waiting for random intervals
  • No grace / retry logic — A failed job is silently skipped until the next random interval
  • No manual trigger — Operators cannot say "run X job now" without restarting the server

Additionally, two jobs (scheduleFeaturedArticleSelection, scheduleHoroscopeGeneration) use node-cron directly, creating a split personality — some jobs are cron-based, some are timer-based — with no unified abstraction.

Goal

  1. Create a single "Absolute Clock" registry that owns all scheduled jobs
  2. Convert all random-interval jobs to deterministic cron expressions
  3. Provide observability — last-run, next-run, status, error count per job
  4. Support graceful error handling — configurable retries, backoff, circuit-breaker
  5. Expose a management API to list, trigger, pause, and resume jobs at runtime
  6. Preserve backward compatibility — existing node-cron jobs migrate cleanly

Implementation Details

1. Migrating Cron Jobs to the Registry

Instead of calling cron.schedule directly inside each job file, each job exports a handler function and registers itself with the Absolute Clock:

// featuredArticleSelection.ts — before
import cron from 'node-cron';
cron.schedule('0 0 * * *', async () => { /* ... */ });

// featuredArticleSelection.ts — after
export async function selectDailyFeaturedArticle(): Promise<void> {
  // ... same body, no cron.schedule here
}

2. Absolute Clock Engine

import cron from 'node-cron';
import { debugLog, debugWarn } from '../utils/debugLogger.js';

const registry = new Map<string, JobDefinition>();

export function registerJob(def: JobDefinition): void {
  if (registry.has(def.name)) {
    throw new Error(`Job "${def.name}" is already registered`);
  }
  registry.set(def.name, def);
  cron.schedule(def.cron, async () => {
    await executeWithRetry(def);
  });
  debugLog(`📅 Registered job: ${def.name}  (cron: ${def.cron})`);
}

Alternative Approaches Considered

1. Bull / BullMQ (Rejected)

Redis-backed job queue. Overkill for a single-process Node app. Adds Redis dependency and operational complexity.

2. agenda.js (Rejected)

MongoDB-backed scheduler. Same concern — external DB dependency for what node-cron handles natively.

3. Keep startRandomInterval (Rejected)

The randomness adds zero value for content generation. It makes debugging harder and observability impossible.

4. Linux cron (Rejected)

External process management. Loses Node.js context, cannot share in-memory state, harder to deploy in containers.
